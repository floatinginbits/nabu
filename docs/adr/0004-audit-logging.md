# 0004 — Best-effort audit logging at the service layer

- **Status:** Accepted
- **Date:** 2026-07-18

## Context
`ARCHITECTURE.md` names audit logging a day-one data-model concern, on the grounds that retrofitting an audit trail is painful and usually incomplete, and `docs/conventions/data-model.md` requires every state-changing service method to write an audit row. Neither says what happens when that write fails, what a row is allowed to contain, or where in the stack it is produced. Those are the decisions that determine whether the trail is trustworthy and whether it can take the product down, so they are worth settling before there are fifteen mutating methods rather than three.

Three constraints shape the answer:

- **There is no transaction to join.** Every repository is constructed as `sqlcgen.New(pool)`; nothing in the codebase threads a `pgx.Tx` from a service into a repository, and the one place a transaction exists (`auth.PostgresRefreshRepository.Rotate`) opens and commits it entirely inside the repository. Making audit writes transactional means introducing transaction plumbing across every domain — a change to every repository signature — as a prerequisite for a feature whose value does not depend on it.
- **The middleware layer cannot produce a useful row.** Middleware sees a method, a path, and a status. It does not know that `POST /api/v1/tasks` created the task with *this* id in *that* project, and it certainly cannot tell a failed login from a successful one before the handler runs.
- **Metadata is a serialization boundary in disguise.** `user.User` carries `PasswordHash` with an explicit comment keeping it out of DTO mapping. An audit `metadata` column is a DTO mapping wearing different clothes, and it is the worst possible destination for a leak: unindexed, unredacted, never expiring, and read by whoever later builds an admin UI over it.

## Decision

**Audit rows are written by domain services, after the state change, outside any transaction.** `audit.Recorder.Record(ctx, Entry)` **returns no error**. This is the load-bearing part of the design: a caller handed an error would have to decide whether to fail an operation that already succeeded, and every caller would make that decision differently. Removing the error removes the question. The recorder logs its own failures via `slog.Error`.

The consequence is stated plainly rather than hedged: **a crash between the state change and the audit insert loses a row**, and so does an audit-table outage. That is the accepted trade. The alternative — audit failure fails the request — means a full `audit_logs` disk or a bad index takes down task creation, which is a worse outcome than an incomplete trail for a tool that is not a compliance system of record. Nabu is not making a non-repudiation claim; it is answering "who changed this task".

**`Entry.Metadata` is `map[string]any`, never `any` and never a domain struct.** Every entry's metadata is built by a per-entity allowlist function living in the package that owns the entity (`task.auditMetadata`, `auth.loginAuditMetadata`). Passing `any` and letting `json.Marshal` walk a domain struct is forbidden, because it makes the set of persisted fields a property of the struct rather than a decision: adding `PasswordHash`-adjacent data to a domain type would silently start writing it to an append-only column. The hard rule is: **never a password or its hash, never a refresh-token plaintext or hash, never a JWT.**

**Encoded metadata is capped at 8 KB and truncated to `{"_truncated": true}` rather than dropped.** Without a cap, a pasted 2 MB description makes every update a large JSONB write; more importantly, because `Record` swallows errors, an insert that failed on size would produce *no audit row and a successful response* — an audit-suppression path an attacker can trigger deliberately by making the payload large. Truncating keeps the row, which is what matters: knowing the action happened beats knowing its details.

**The failure log never contains metadata.** It carries `action`, `entity_type`, `entity_id`, `actor_id`, and the error. Redacting a field at the column and then printing it into a log pipeline that ships off the host redacts nothing.

**Cancellation does not abandon the write.** `Record` runs on `context.WithoutCancel(ctx)` with its own 5-second timeout: the row describes work that already committed, so a client hanging up mid-response must not erase the record of it, and the timeout keeps that from becoming an unbounded write.

**Schema is complete on arrival (migration `00006`).** `actor_id` (nullable, `ON DELETE SET NULL`), `org_id NOT NULL`, `project_id` (nullable), `action`, `entity_type`, `entity_id` (nullable), `metadata jsonb`, `created_at`, plus an index on `(org_id, created_at DESC)`. `actor_id` is nullable and survives its user because an audit row that is deleted along with the account it describes is not an audit row, and because an unauthenticated event — a failed login — has no user at all. `org_id` and `project_id` are present now rather than after the fact because "who changed what in this project" is the first query anyone writes against this table, and `data-model.md` rules out stuffing scope into the JSONB.

**Auth events pass their actor explicitly.** Login, logout, and failed login run *unauthenticated*, so there is no `actor.Actor` in the context; `auth.Service` sets `Entry.ActorID` from the authenticated user it just resolved (or leaves it null), and takes its `OrgID` from the same startup-resolved singleton the auth middleware stamps onto every session (ADR-0005) — never from the request. The context-actor invariant applies only behind `requireAuth`. `RefreshRepository.RevokeFamilyByHash` now returns the user whose family it revoked, so `auth.logout` names a subject; a logout that revoked nothing (a stale cookie replayed) records nothing rather than writing an actorless row for every expired session a browser presents.

**`ARCHITECTURE.md`'s day-one `NotificationService` is cut, and this ADR is where that is recorded.** `ARCHITECTURE.md` lists a `NotificationService` interface with a v1 no-op implementation as a schema/extensibility point built in from day one; issue #16 removed it from the M2 scope as exactly the speculative abstraction `CLAUDE.md` forbids — an interface with a do-nothing implementation and no caller, whose only real function would be to appear in a constructor signature. That call is right, but a source-of-truth document cannot be overridden by a checkbox in an issue, so `ARCHITECTURE.md`'s Data model section is amended in the same change that lands this ADR. The reasoning is worth keeping because audit logging is the abstraction that *did* survive the same test: it has real callers and a real implementation on day one, which is the difference. Nothing about the cut forecloses notifications later — adding an interface when a notifier exists is a smaller change than carrying an empty one through v1.

## Consequences
Domain services take a `Recorder` by constructor injection and call it after a successful state change; nothing else in the stack knows audit exists. Because `Record` cannot fail, it cannot complicate a service method's error handling, and the audit call is a single unconditional statement rather than a branch. The `audit_logs` schema needs no follow-up migration when RBAC and task CRUD land — they add actions, not columns.

### Limitations and risks
- **The trail is incomplete by construction.** A process crash in the window between commit and insert, or an audit-table outage, silently loses rows. Nothing in the system detects the gap: an action that was never audited is indistinguishable from an action that never happened.
- **A swallowed error is only as good as the log.** If the failure log is not monitored, audit writes can be failing continuously with no user-visible symptom — the deployment looks healthy while the trail quietly stops.
- **The redaction rule is a convention with a hole.** It is enforced by review and by tests, not by a type: `map[string]any` accepts a bcrypt hash just as happily as a title. The risk is not today's two builders, it is the fifth one written by someone who copies a struct field list.
- **Truncation hides exactly the entries most worth reading.** An oversized document is often the interesting one, and `{"_truncated": true}` loses all of it rather than the excess.
- **`audit_logs` grows without bound.** There is no retention policy, no partitioning, and no prune job; on a busy instance this becomes the largest table in the database.
- **Auth's `orgID` is constructor state.** It is resolved once at startup, so a deployment that ever holds more than one org would attribute every login to the wrong one until auth resolves the org per session — the same load-bearing assumption ADR-0005 already carries.

### Mitigations
- **The redaction rule is tested at the `Recorder` boundary, not over today's builders.** `internal/audit/audittest` installs a recording `Recorder` in the `task` and `auth` suites that marshals *every* entry any test produces and fails on a denylist (`password`, `password_hash`, bcrypt `$2a$`/`$2b$` prefixes, the `eyJ` JWT prefix, `token_hash`, `refresh_token`, `secret`). A per-builder assertion could not fail for a builder added next quarter — which is the precise risk this ADR exists to control — whereas any new audited action written by a test picks the check up for free.
- Failure logging is asserted, not assumed: an integration test drives a real foreign-key violation through `PostgresRecorder` and requires both that the operation still succeeds and that the emitted log carries no metadata.
- Truncation and the unencodable-value fallback are covered by table tests, including a round trip proving the marker is valid JSONB — a cap that made the insert fail would defeat its own purpose.
- **Leading indicator that this decision has failed:** a `Record` call passing a domain struct or an `any`, a metadata key that is not a literal in an allowlist function, or a `Recorder` implementation that returns an error. Any of those means the boundary has eroded and the redaction rule is no longer checkable.
- Retention and the eventual prune/partition strategy are deferred deliberately, alongside the refresh-token prune sweep already owed to ADR-0003 — the same worker slice.

## Alternatives considered
- **Transactional audit writes in the same transaction as the state change.** The strongest guarantee: the row and the change are atomic, and the trail cannot silently miss an event. Rejected for v1 on cost and blast radius — it requires transaction plumbing through every repository signature in the codebase before the first audited action can land, and it inverts the failure mode so that an audit-side problem fails user-visible writes. Reconsider if Nabu ever makes a compliance claim that requires the trail to be complete rather than merely useful; the `Recorder` interface is the seam where a transactional implementation would attach.
- **Audit in HTTP middleware.** Cheap and uniform — one place, every endpoint, impossible to forget. Rejected because the resulting row is `POST /api/v1/tasks → 201` with no entity id, no project, and no way to tell a failed login from a successful one. The forgettability that service-layer placement introduces is real and is what the RBAC coverage test pattern (phase 3) exists to catch.
- **Asynchronous audit via a queue or a buffered channel.** Removes the insert from the request path entirely. Rejected as premature: it trades a synchronous best-effort write for an in-memory buffer that loses *more* rows on crash, and it needs the worker slice that does not exist yet. The 5-second timeout bounds the synchronous cost in the meantime.
- **Postgres triggers writing audit rows from the DDL layer.** Complete by construction — nothing in the application can forget. Rejected: a trigger sees the row, not the intent (it cannot distinguish "assigned" from "reassigned", and knows nothing about the session actor without per-transaction `SET LOCAL` plumbing on a pooled connection), and it puts business semantics in a place `sqlc` cannot see and tests cannot easily reach.
- **A single generic `metadata any` marshalled by the recorder.** Convenient, and every caller would use it. Rejected as the exact mechanism by which a password hash reaches an append-only column — the convenience is inseparable from the hazard.
